### 플랫폼 개요

프론트엔드 코드가 대부분 완성되었으며, 현재 코딩 교육용 웹 애플리케이션은 다음과 같은 주요 페이지로 구성됩니다:

- 로그인 페이지
- 챌린지 페이지(문제 목록 확인)
- 스튜디오 페이지(블록 코딩 및 Python 변환)
- 프로필 페이지(수집한 블록 확인)
- 어드민 페이지(문제 관리 및 제출물 평가)

### 주요 기능 설명

- **스튜디오 페이지**: 블록 코딩을 통해 Python 코드 생성 가능, 코드 실행 및 다운로드 기능 포함
- **챌린지 페이지**: 사용자가 풀 수 있는 문제 목록 표시, 난이도별 문제 제공 예정
- **프로필 페이지**: 사용자가 수집한 블록과 아직 수집하지 못한 블록 표시, 'All', 'Collected', 'Missing' 탭으로 구분
- **어드민 페이지**: 문제 관리, 사용자 제출물 평가, QR 코드 생성 기능 예정

### QR 코드 및 블록 수집 메커니즘

- 일부 코딩 블록은 QR 코드를 스캔해야 수집 가능하도록 설계
- 블록 관리 방안에 대한 논의:
    - 모든 블록을 QR로 만들지 않고 일부 핵심 블록만 QR로 제공
    - 한 QR 코드에 여러 블록을 포함시키는 방안 고려
    - 어드민 페이지에서 QR로 제공할 블록을 관리자가 선택 가능하게 구현
    - 기본 블록은 처음부터 사용 가능하고, QR로 수집해야 하는 블록은 미션처럼 제공

# **1) 한줄 요약**

- 채점은 1차 수동(관리자가 코드/출력 확인 후 점수).
- 용어/UX(“미싱” 등) 다듬기, 하단 탭/스캐너 버튼 위치 등 UI 디테일은 추후 확정.

# **2) 현재 구현/구상 화면 & 기능**

**홈 (Home)**

- 랜딩 페이지. 작업 미착수, 마지막 단계에서 디자인/카피 확정 예정.

**챌린지(Challenges)**

- 학생이 풀 수 있는 문제 목록 페이지.
- 난이도별/주차별 노출 상정(지민 담당자가 조절).

**스튜디오(Studio)**

- 문제 풀이 + 블록코딩 화면.
- 블록 → Python 코드로 변환 및 실행(Run) 가능.
- 다운로드: .py 파일 저장 (대부분 변환 정상 예상, 전체 케이스 테스트는 미완).
- Submit(제출): 풀이 제출 → 어드민에서 채점.
- Save(임시저장): 진행 중 저장(우선순위 낮음, 필수 아님).

**프로필(Profile)**

- 보유 가능한 전체 블록(All), 획득(Collected), 미획득(to-collect) 탭 구조.
- QR 스캔으로 블록 획득 시 Collected로 이동.
- 상단에 보유 블록 수/주요 진행도 등 요약 카드.

**어드민(Admin)**

- 제출물 목록/상세 리뷰 → 점수 부여(수동 채점).
- 문제(Questions) 관리:
    - 웹에서 직접 생성/수정 or CSV 업로드로 일괄 반영(운영 편의성 고려).
- 
- 블록/QR 관리(추가 예정):
    - 블록 사용 여부 On/Off 토글
    - QR 생성: 단일/복수 블록을 한 QR로 묶음 생성 옵션 검토
    - 주기적(예: 2주) QR 세트 교체 운영 시나리오 지원
- 

# **3) QR·블록 운영 안(논의 내용 정리)**

- 전체 블록을 전부 QR화하지 말 것(과다/운영 난점).
- 핵심 블록만 선택하여 QR로 배포 → 학생이 스캔 시 해당 블록 획득.
- 나머지(핵심 외) 블록은 기본 제공(문제 풀이 가이드/난이도 설계 용이).
- 운영 예시:
    1. *관리자(지민)**가 주기별(예: 2주)로 필요한 핵심 블록만 체크
    2. 체크된 블록을 QR로 생성(단일 또는 묶음)
    3. 학생은 해당 QR을 스캔하여 필수 블록 확보
- 
- 논의 포인트
    1. 한 QR당 여러 블록을 묶어 배포하면 “한번에 너무 많이 획득” → 챌린지성이 약화될 수 있음 → **소수(예: 3~5개)**로 묶는 방안 검토
    2. QR의 부착 위치/힌트 제공 방법 필요(현장 운영 UX)
- 

# **4) 용어·UX 관련 메모**

- “미싱(missing)” 용어는 오해 소지 → 대체안:
    - “필요 블록”, “획득 대기”, “아직 못 얻은 블록”, “To Collect”
- 
- 하단 탭 vs 상단 네비: 모바일 스캔 편의성 고려해 하단 탭 유지 의견 강함(추후 확정).
- 스캐너 버튼 위치: 탭 영역 고정 버튼 vs 상단 액션버튼 등 별도 진입 버튼 필요(쉽게 접근 가능하게).

# **5) 결정/합의 사항**

- 현 구조(로그인/Challenges/Studio/Profile/Admin)로 계속 개발 진행.
